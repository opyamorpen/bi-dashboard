# BI 仪表盘分析插件技术实现方案

## 1. 技术目标

基于 ONES Open Platform 1.0 实现一个 `.opk` 插件，为 ONES 团队提供 BI 仪表盘、数据集、指标配置、图表卡片、下钻明细、权限控制、刷新缓存、订阅导出、告警和审计能力。

本方案严格按 Open Platform 1.0 插件模型设计：

- 使用 `ones create` / `npx op` 创建、调试、打包。
- 使用 `config/plugin.yaml` 声明模块、接口、OAuth scope、事件、实体存储和能力。
- 前端模块位于 `web/src/modules/*`。
- 后端接口、事件、生命周期函数位于 `backend/src`。
- 交付物为 `.opk`。
- 不使用 Open Platform 2.0 的 `.opkx`、`opkx.json`、Hosted API、Extensions 或 Web SDK。

## 2. 实现结论

### 2.1 总体可行性

整体按 1.0 插件可实现，但需要分级落地：

- P0 可作为首版核心交付：仪表盘、数据集、指标维度、卡片布局、基础图表、筛选查询、下钻明细、基础权限、刷新缓存、异常状态。
- P1 建议作为第二阶段：发布版本、模板、全局筛选、图表联动、分享订阅、导出、审计和资产统计。
- P2 建议作为增强阶段：告警、高级图表、AI 洞察、复杂数据血缘和智能推荐。

### 2.2 关键边界

- 数据获取依赖 OpenAPI、插件 OpenAPI 调用能力，以及客户环境中可用的 ONESQL 或等价查询能力。
- 用户查看数据时优先使用用户身份查询，确保继承 ONES 数据权限。
- 后台缓存、订阅、告警等场景如需管理员身份聚合数据，必须在展示、导出和推送前按接收人权限重新过滤。
- 插件 API 单次返回数据需控制在 16 MB 以内，明细和导出必须分页或异步处理。
- 1.0 实体存储每个实体最多 32 个属性，复杂配置使用 `text` JSON 存储，关键筛选字段使用 `string` / `integer` 建索引。
- 后端运行在 node-host sandbox 中，不依赖全局内存保存状态。

## 3. ONES 1.0 能力映射

| 功能域 | 1.0 能力 | 实现方式 |
| --- | --- | --- |
| 插件入口 | Slot / Navigation Slot | 新增 BI 仪表盘页面入口和配置页入口 |
| 仪表盘前端 | Web slot module | React 单页应用承载列表、详情、编辑器 |
| 前后端通信 | `apis` 注册插件接口 | `addition` API 为前端提供业务接口 |
| ONES 数据读取 | OpenAPI / 插件 OpenAPI 访问 | `@ones-op/fetch`，按用户或管理员身份读取 |
| 数据集与仪表盘配置 | Entity storage | 保存仪表盘、卡片、数据集、权限、版本等配置 |
| 快照/导出文件 | Object storage | 保存导出文件、快照文件或临时文件 |
| 刷新、订阅、告警 | Timer ability | 定时刷新缓存、发送订阅、计算告警 |
| 数据变更感知 | Events | 监听项目、工作项、迭代、字段等变化后失效缓存 |

| 通知 | ONES notice / external notification | 订阅、告警结果通知 |

## 5. `config/plugin.yaml` 设计要点

### 5.1 基础信息

- `service.app_id`：首次发布后保持不变。
- `service.version`：每次升级递增。
- 插件类型：建议团队级插件；若要求跨团队统一资产和组织级权限，需要评估组织级插件。
- 授权策略：建议首版选择 `standard`，便于明确 OAuth scope 和权限边界。

### 5.2 前端模块

建议声明两个模块：

- `bi-dashboard-app`：BI 仪表盘主页面。
- `bi-dashboard-config`：插件配置页面。

入口可选方向：

- 独立侧边栏菜单入口：适合完整 BI 工作台。
- 效能管理导航入口：适合定位为效能分析增强。
- 项目内入口：适合强项目上下文仪表盘。

最终 slot 类型需结合目标 ONES 版本和客户入口要求，从 navigation / module slot 中确认。

### 5.3 后端接口

使用 `apis` 注册 `addition` 类型接口，供前端模块调用：

```yaml
apis:
  - type: addition
    methods: [POST]
    url: /bi/query
    function: biQuery
  - type: addition
    methods: [POST]
    url: /bi/detail
    function: biDetail
  - type: addition
    methods: [POST]
    url: /bi/export
    function: biExport
```

前端调用接口时需要携带：

- `Ones-Plugin-Id`
- `Ones-Check-Point`
- `Ones-Check-Id`

### 5.4 OAuth Scope

按最小权限申请，初始建议：

- `read:project:project`
- `read:project:issue`
- `read:project:issueType`
- `read:project:issueStatus`
- `read:project:issueField`
- `read:account:user`
- `read:account:department`
- `read:account:department-member`
- 工时相关 scope 根据客户环境和实际接口补充。

如需要写评论、创建工作项、修改工作项，不纳入本 BI 插件首版 scope。

## 6. 数据模型设计

### 6.1 实体设计原则

Open Platform 1.0 实体存储存在属性数量、索引和字符串长度限制，因此采用“关键字段索引 + JSON 配置体”的模式：

- 用 `string` / `integer` 保存高频查询字段。
- 用 `text` 保存复杂配置 JSON。
- 不在升级中修改或删除既有属性和索引，只追加新字段。
- 所有实体包含 `team_uuid`、`created_by`、`created_at`、`updated_by`、`updated_at`、`is_deleted`。

## 8. 查询引擎设计

### 8.1 查询流程

1. 前端提交卡片查询请求，包含 `dataset_uuid`、指标、维度、筛选、排序、分页、时间口径。
2. 后端校验用户对仪表盘和卡片的插件权限。
3. 后端读取数据集定义和卡片配置。
4. 后端将全局筛选器、卡片筛选器、下钻条件合并为统一查询 AST。
5. 权限服务追加当前用户可访问数据范围。
6. 查询引擎选择数据提供器：
   - 优先使用 ONESQL 或客户环境可用查询能力处理聚合。
   - 无 ONESQL 时，使用 OpenAPI 分页拉取后在插件后端聚合。
7. 后端执行聚合、排序、Top N、时间补齐、隐藏 0 值等处理。
8. 返回图表数据、口径说明、查询耗时、更新时间和异常信息。

### 8.2 数据提供器

#### OpenAPI Provider

适合：

- 元数据读取。
- 中小数据量聚合。
- 明细表分页。
- 权限继承要求高的实时查询。

限制：

- 大规模聚合性能受接口分页和限流影响。
- 部分历史状态和关联统计需要额外接口支持。

#### ONESQL Provider

适合：

- 大数据量聚合。
- 多维分组。
- 复杂筛选。
- 趋势统计。

限制：

- 是否可用取决于客户环境和开放能力。
- 字段、表结构和权限模型需专项确认。

### 8.3 聚合能力

首版支持：

- count
- distinct count
- sum
- avg
- max
- min
- ratio
- cumulative
- month cumulative
- compare with previous period
- compare with same period last year

多选字段拆分聚合：

- 查询时将多选字段拆分为多个选项命中。
- 同一工作项可在多个选项中分别计数。
- 图表口径中必须展示该说明。

### 8.4 时间口径

- 单指标趋势图可按完成时间、创建时间、更新时间等业务时间字段展示。
- 多指标趋势图优先按自然时间轴对齐。
- 支持自然日、自然周、自然月。
- 季度、年度作为后续增强。

## 9. 权限设计

### 9.1 插件资产权限

仪表盘资产权限由 `bi_dashboard_acl` 控制：

- owner：管理全部配置、权限、发布、删除、转让。
- editor：编辑仪表盘、卡片、筛选器、发布草稿。
- viewer：查看仪表盘、下钻明细。

导出权限可作为独立开关存入仪表盘配置。

### 9.2 ONES 数据权限

原则：

- 前台查看查询尽量使用 `FetchAsUser`，继承用户自身权限。
- 后台缓存如使用管理员身份，缓存结果不能直接对所有人复用，必须按用户权限二次过滤或按权限范围生成缓存。
- 明细和导出必须再次校验用户数据权限。

### 9.3 管理员权限

插件管理员可以：

- 修改插件配置。
- 查看运行日志。
- 查看慢查询。
- 管理全局数据源策略。

插件管理员不自动获得所有仪表盘业务数据查看权，除非其在 ONES 中本身具备对应数据权限。

## 11. 数据刷新、缓存与性能

### 11.1 缓存策略

缓存 key 由以下信息生成：

- `team_uuid`
- `dashboard_uuid`
- `card_uuid`
- `dataset_version`
- `query_json`
- `filter_json`
- `viewer_permission_hash`

缓存粒度：

- 元数据缓存：项目、字段、状态、成员等。
- 卡片查询缓存：聚合结果。
- 明细查询不默认缓存，仅分页查询。

### 11.2 并发控制

- 仪表盘加载时按卡片分批请求。
- 同一仪表盘默认并发 4 到 8 个卡片查询，可配置。
- 100 张卡片场景采用首屏优先、可视区域优先、延迟加载策略。
- 卡片超时不阻塞其他卡片展示。

### 11.3 超时与重试

- 单卡片查询设置超时时间。
- 元数据请求失败可重试。
- 聚合查询失败记录日志并返回卡片级错误。
- 后台定时任务失败不影响前台实时查询。

### 11.4 16 MB 限制处理

- 聚合接口只返回图表所需数据。
- 明细接口必须分页。
- 导出接口创建异步任务，不在单次接口中返回大文件。
- 大文件使用对象存储保存，前端轮询任务状态后下载。

## 12. 发布、版本与升级

### 12.1 仪表盘发布版本

- 草稿数据保存在 `bi_dashboard.config_json` 或独立 draft 配置中。
- 已发布版本保存至 `bi_dashboard_version.snapshot_json`。
- 查看态默认读取已发布版本。
- 编辑态读取草稿版本。
- 回滚时将历史 `snapshot_json` 写回草稿或发布版本。

### 12.2 插件升级

升级规则：

- 保持 `service.app_id` 不变。
- 递增 `service.version`。
- 保留 `config/upgrade.yaml`。
- 实体存储只追加字段和索引，不修改或删除既有字段。
- 新字段对历史数据返回 `null` 时，业务逻辑需提供默认值。

## 13. 订阅、导出、告警

### 13.1 订阅

实现方式：

- 前端创建订阅配置。
- 后端保存到 `bi_subscription`。
- Timer 定期扫描到期订阅。
- 按订阅人权限查询数据。
- 生成快照或导出文件。
- 通过可用通知渠道发送。

### 13.2 导出

实现方式：

- 小数据量表格可同步导出。
- 仪表盘 PDF、图片和大数据 Excel 使用异步任务。
- 任务状态保存到 `bi_export_task`。
- 文件保存到对象存储。
- 下载前校验用户导出权限。

### 13.3 告警

实现方式：

- 告警规则保存到 `bi_alert_rule`。
- Timer 定期计算告警条件。
- 告警计算使用规则配置的卡片查询逻辑。
- 命中后写入告警历史并发送通知。
- 支持静默窗口和重复告警合并。

## 14. 事件与缓存失效

建议监听以下事件族，具体 eventType 需按目标 ONES 版本文档确认：

- 工作项创建、更新、删除。
- 工作项字段变更。
- 项目创建、更新、成员变更。
- 迭代创建、更新。
- 项目字段、团队字段、状态、工作项类型变更。

处理原则：

- 事件处理必须按 `eventID` 幂等。
- 不在事件处理函数中执行大规模重算。
- 事件只标记缓存失效或写入刷新队列。
- 真正刷新由定时任务或前台请求触发。

## 15. 审计与运维

### 15.1 日志类型

- 操作日志：创建、编辑、删除、复制、分享、导出、转让。
- 查询日志：查询耗时、命中缓存、数据量、失败原因。
- 导出日志：任务创建、完成、失败。
- 通知日志：订阅和告警发送结果。
- 配置日志：插件配置变更。

### 15.2 慢查询

慢查询记录字段：

- `query_id`
- `dashboard_uuid`
- `card_uuid`
- `operator_uuid`
- `duration_ms`
- `provider`
- `query_summary`
- `created_at`
- `error_message`

管理员可按耗时、卡片、数据集、时间范围筛选。

## 16. 安全设计

- 前端不保存管理员凭证。
- `config/local.yaml` 和 `config/ci-deploy.yaml` 不提交真实密钥。
- 所有后端接口校验 `team_uuid`、用户身份、插件权限和数据权限。
- 所有导出任务校验导出权限。
- 日志中不记录敏感字段明文。
- 卡片配置中的查询条件需要白名单校验，避免注入式查询。
- 对 ONESQL 或等价查询能力的请求必须通过查询构造器生成，不拼接未校验字符串。

## 19. 验证方案

### 19.1 功能验证

- 创建仪表盘。
- 添加 10 种基础图表卡片。
- 配置数据集、指标、维度、筛选。
- 调整卡片尺寸和布局。
- 下钻到明细。
- 明细跳转工作项详情。
- 分享给编辑者和查看者。
- 转让 owner。
- 删除和恢复仪表盘。

### 19.2 权限验证

- owner 可编辑、转让、删除。
- editor 可编辑但不可转让。
- viewer 仅可查看。
- 无权限用户不可访问仪表盘。
- 用户看不到其在 ONES 中无权访问的数据。
- 无导出权限用户不可导出。

### 19.3 数据验证

- 项目、迭代、工作项、状态、字段、成员数据正确展示。
- 自定义字段可作为维度或筛选条件。
- 多选字段拆分聚合结果符合口径。
- 多指标趋势图时间口径符合配置。
- 缺失字段触发配置异常提示。

### 19.4 性能验证

- 单仪表盘 100 张卡片加载。
- 大数据量明细分页。
- 查询缓存命中。
- 慢查询记录。
- 单接口响应不超过 16 MB。
- 单卡片失败不影响其他卡片展示。

### 19.5 升级验证

- 升级后原仪表盘、数据集、卡片、权限仍可访问。
- 新增实体字段历史数据默认值处理正确。
- `service.app_id` 未变化。
- `config/upgrade.yaml` 生效。
