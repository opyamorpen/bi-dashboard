# ONES 固定报表中心产品方案

## 1. 产品定位

本插件从“通用自助 BI 仪表盘”调整为“ONES 固定报表中心”。核心目标不是让每个用户在页面上无限自由搭建 BI，而是让交付人员或管理员可以根据客户描述，快速生成、确认并发布一套固定报表。

固定报表由明确的数据范围、分析指标、分析维度、筛选条件、图表类型和布局组成。报表创建后优先读取已刷新的快照数据，查看页面不再默认每次全量查询 ONESQL，从而降低大数据量环境下的查询压力。

## 2. 目标用户

- 项目经理：查看项目进展、工作项数量、负责人分布、状态分布等固定报表。
- PMO / 管理者：查看跨项目质量、效率、风险和交付趋势报表。
- 插件管理员 / 交付人员：根据客户报表需求快速配置并发布固定报表。
- 普通查看者：只查看已发布的报表结果，不参与底层查询配置。

## 3. 核心业务流程

### 3.1 AI 生成报表草稿

用户输入自然语言需求，或上传包含报表描述、截图、表格样例的图片。插件调用可配置的 OpenAI 兼容模型，将需求转换为结构化报表草稿。

AI 草稿只包含报表标题、说明、数据范围、筛选条件、卡片列表、指标、维度、图表类型和推荐布局。AI 不直接生成任意 SQL，也不直接执行查询。

### 3.2 用户确认报表草稿

系统展示 AI 生成的草稿内容，用户确认或调整后才能创建固定报表。确认前不会写入正式报表，也不会触发数据查询。

### 3.3 创建固定报表

用户确认草稿后，系统创建固定报表和卡片配置。卡片只允许使用插件支持的字段、聚合方式和图表类型。

首期支持的字段：

- `uuid`：工作项 ID
- `title`：标题
- `issue_type`：工作项类型
- `status`：状态
- `assignee`：负责人
- `priority`：优先级
- `project_uuid`：项目
- `created_at`：创建时间

首期支持的图表：

- 数字指标卡
- 柱状图
- 饼图
- 环形图
- 表格

### 3.4 手动刷新快照

报表创建后，用户在详情页点击“刷新快照”。插件按卡片配置查询 ONES 工作项数据，聚合后保存为报表快照。

查看态优先读取快照数据。已有快照时不自动重复查询全量数据。没有快照时展示“尚未刷新”，提示用户手动刷新。

### 3.5 查看固定报表

普通查看者进入报表后只看到卡片内容、图表和刷新状态。查看态不显示网格背景、拖拽控制、ONESQL 数据源说明和耗时信息。

编辑态用于调整报表标题、说明、卡片位置和尺寸。高级编辑保留现有卡片配置能力，用于交付人员处理特殊场景。

## 4. 功能范围

### 4.1 P0 首期范围

- AI 服务配置：配置 OpenAI 兼容 `base_url`、`api_key`、`model`、是否支持图片。
- AI 报表草稿生成：支持文本输入和图片输入。
- 草稿校验：只接受白名单字段、图表类型和聚合方式。
- 草稿确认创建报表：确认后生成固定报表和卡片。
- 报表列表：查看、新建、删除固定报表。
- 报表详情：查看快照数据、进入编辑态、调整卡片布局。
- 手动刷新快照：逐卡片查询并保存聚合结果。
- 快照优先展示：查看态优先读取快照，避免每次打开报表都全量查询。
- 高级编辑：保留现有手动卡片配置能力。

### 4.2 P1 增强范围

- 固定报表模板库：内置常见项目管理、缺陷质量、成员工作量、风险分布模板。
- 定时刷新快照：支持按天、按周、按月刷新。
- 报表复制：基于已有报表快速复制并调整数据范围。
- 导出：导出图片、PDF 或 Excel。
- 订阅：按固定周期推送报表结果。
- 更多图表：折线图、堆叠柱状图、趋势表。

### 4.3 P2 远期范围

- 报表市场：沉淀客户常用报表模板。
- 告警：指标超阈值后通知负责人。
- 多模型支持：按团队配置不同 AI provider。
- 深度 ONESQL 优化：服务端预聚合、分区快照、增量刷新。
- 自然语言解释：针对快照结果生成摘要和风险提示。

## 5. 权限与安全边界

- 插件内的报表权限只控制报表资产，不扩大用户在 ONES 中的数据权限。
- 刷新快照应使用当前用户可访问的数据范围，避免把无权数据固化到快照。
- AI 配置中的 `api_key` 不在前端明文返回。
- AI 只生成结构化草稿，不允许直接生成或执行任意 ONESQL。
- 报表字段和查询条件必须经过白名单校验。

## 6. 性能策略

客户生产环境可能存在百万级工作项数据。首期不再把“每次打开报表实时全量查询”作为默认路径，而采用快照优先：

- 查看报表：读取插件快照实体。
- 刷新报表：由用户显式触发。
- 后续增强：通过定时任务、增量刷新和预聚合减少全量扫描。

该策略可以先满足“报表必须基于真实全量数据”的目标，同时避免用户每次查看都对 ONESQL 造成高压查询。

## 7. 成功标准

- 用户能通过自然语言或图片生成结构化报表草稿。
- 用户确认后能创建固定报表。
- 报表卡片能保存并展示真实快照数据。
- 查看态不会自动触发全量 ONESQL 查询。
- 编辑态和查看态边界清晰。
- 后续新增固定报表时，主要工作变成“描述需求、确认草稿、刷新快照”，而不是逐项手动搭建 BI。
